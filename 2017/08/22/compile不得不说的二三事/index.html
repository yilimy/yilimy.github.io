compile不得不说的二三事

发表于 2017-08-22 | 阅读次数
compile项目时要注意依赖和版本号

项目中要用到android对PDF进行签名，需要用到Itext家的jar包 Itext家的官网: http://itextpdf.com 如果是一般项目按照官网上的配置,一般是不会有什么差池的

然而运行签名(“MakeSignature.signDetached”)是怎么都跑不通，崩啊崩就像我的心跳，不停地崩，摊手~ 其实按照git上的描述： The main iText release contains:

itextpdf-x.y.z.jar: the core library itext-xtra-x.y.z.jar: extra functionality (PDF 2!) itext-pdfa-x.y.z.jar: PDF/A-related functionality xmlworker-x.y.z.jar: XML (and HTML) functionality

我只需要compile core包就可以了，然而Gradle一直提示我，XXX方法未找到，XXX构造器没有 不死心的我就把剩下的包全部compile进去，还以为官方诚不我欺呢，P 接着就调整各个包的版本号，恩，少方法嘛，包多多益善，版本多多尝试，看我调通了把你们丫一个个踢出门 未果… 可能是报应，Gradle控制台爆发了“AGPBI”洪流，原本完好的项目，现在一次都跑不通了 持续崩溃中… 说debug文件夹下有jar包冲突，删删删~ 未央！ 纠缠半晌发现是libs下删除了，没有做最后的确认“do refactor”，那货死皮赖脸的留着项目中呢 干掉之！ 久违的项目跑通，接着怼。 刚才到哪了？…哦，换包换版本哈 唔，添加“xtra”，“pdfa”，“xmlworker”这三个小崽子中的任何一个都在编译的时候给我蹦出个 "org.apache.commons imaging 1.0-SNAPSHOT"compile的异常

可气的是这个imaging包居然抓不下来，怀疑人生中… 第三方依赖出问题了，你还有理了！干掉之 想想不对啊，这三小崽子，有依赖，没理由“itextpdf”这个core包没有啊 在maven库中瞅瞅，https://mvnrepository.com/artifact/com.itextpdf/itextpdf/5.5.12 咦，还真有依赖 再看看项目，bcprov-jdk15on-1.47.jar有，在把bcpkix-jdk15on-1.47.jar加进去 还是报错。 把这两货调到最高版本1.57，奇了怪了，居然成了。再回首，图中的依赖说的是最低1.49吧。 不得不提及为啥会有bcprov-jdk15on-1.47.jar和bcpkix-jdk15on-1.47.jar 这是另一个坑的印记。 java认可的JKS证书和android不认啊，android认“BKS”，要在加密时使用bouncycastle家的provider。 如此有了bcprov和bcpkix，而且加密要用bcprov而不是bcpkix，否则会出现版本问题。 一行热泪。 。。。 网上有说Itext版本更新频繁，以前低版本方法可能不能用，哪知道有这么多坑。 还好是通了，就这样吧，打点一下回家去。
